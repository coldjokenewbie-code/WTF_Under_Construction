## 一、技術規格

### 尺寸
- 搪床、刨床操作機台尺寸皆為**寬80cm×高145cm**（`_context/PRD_2026-05-18_搪床互動機具腳本.md` 第17行；`_context/PRD_2026-05-27_刨床互動機具腳本.md` 第20行）。
- 車床操作機台尺寸為**108×149cm**，列於此處僅供比例參照，非本次三機具之一（`_context/INDEX.md` 第32行）。
- 刨床定稿簡報P3頁另有「130cm」「30cm」局部標註，與整機80×145cm之間的對應關係未說明清楚（`workingfiles/ppt_analysis_output.txt` Slide 3）——**缺口，需業主／原始設計稿釐清**。
- 鑽床操作機台尺寸：僅有語意含混的「1200mm／1550mm，標註對象『鑽床展示櫃』」一組數字，且與PRD詢問之108×149cm換算後不符（`workingfiles/w_boring_structure.txt` 第56-59行）——**缺口，待現場確認**（詳見機具章節鑽床節）。

### 螢幕
- 搪床螢幕原為直式，2026-05-27重構時修正為**橫式16:9**，整合於機台上半部（`walkthrough.md` 第37行）。
- 刨床螢幕形式為**橫式16:9**，設計理由是配合龍門鉋床長型加工特色（`_context/PRD_2026-05-27_刨床互動機具腳本.md` 第21行）。
- 鑽床螢幕形式：草稿逐頁文字未見明確描述——**缺口**。

### 操作機構
- 搪床：雙旋鈕設計——實體大手輪（Z軸向進給）＋左下微調小圓鈕（設定孔徑），另配六角扳手與壁掛架（`workingfiles/boring_analysis_output.txt` Slide 3；`_context/PRD_2026-05-18_搪床互動機具腳本.md` 第13-14行）。
- 刨床：**僅上方手柄（Downfeed handle／下進給螺桿手柄），機台無下方轉輪**，此為業主明確指定之限制（`walkthrough.md` 第14、26行）。
- 鑽床：草稿設計為「進刀操作桿＋鑽削深度設定刻度盤」，但未經業主正式核准，狀態為待確認（`workingfiles/drilling_structure.txt` 第80行；詳見機具章節鑽床節）。

### 電源
- 全repo查無任何互動機台本身的供電規格、電壓、功率或用電安全文件。現有的「馬力」數字（如台工11號7.5hp、台工54號20hp）均為**現地歷史文物本身的動力源規格**，並非互動展示機台的電力需求（`reference/搪床原理與腳本提案.html` 第1303行；`reference/龍門鉋床原理與互動腳本提案.html` 第65行）——**缺口，屬於互動機台工程施工前必須由廠商或業主補齊的項目**。

### 感應
- 現有素材僅描述到「使用者轉動手輪／旋鈕後畫面／數值即時反應」的劇本層級效果，未見任何感測器選型（如編碼器、電位計、微動開關等）、觸發精度、校準方式或機構工程圖（遍查 `_context/`、`reference/`、`workingfiles/` 全文皆無相關關鍵字）——**缺口，是本章節中「劇本可行但無法直接施工」的核心落差**。

### 配色與字體規範（面板設計標準）
- 背景 `#0a0a0a`、面板 `#141414`、強調橘 `#FF5A00`、成功綠 `#00FF88`；正文採黑體，系統／技術標註採JetBrains Mono等寬字體，全系列（含車床、銑床）統一適用（`rules/色彩計畫.md` 第1-6行；`reference/通用機具簡報架構.md` 第82-84行）。

## 二、驗收計畫素材

### 現有自動化驗收工具
repo內以 `find` 指令確認全域僅存在**兩支** `verify_*.py` 腳本，分別對應搪床、刨床兩份定稿簡報，鑽床、車床、銑床皆無對應驗收腳本：

- **`workingfiles/_scripts/verify_boring_ppt.py`**（70行）驗證項目：
  1. 簡報頁數是否恰為10頁。
  2. 禁詞掃描：`banned_words = ["鑽床","鑽削","鑽頭","鑽孔","進刀操作桿","台工 20 號","台工 1510 號"]` 不得出現於搪床簡報（第16行）。
  3. 必要字詞檢查：`required_words = ["台工 11 號","台工 364 號","搪床","精準擴孔"]` 必須出現（第42行）。

- **`workingfiles/_scripts/verify_planer_ppt.py`**（115行）驗證項目：
  1. 簡報頁數是否恰為10頁。
  2. 禁詞掃描（含跨機種殘留字詞，以及「鑽床/鑽削/搪床/搪削」等情境式禁詞，需人工複查是否為科普對比金句之假陽性，第17、24行）。
  3. 必要字詞檢查：`required_words = ["台工 54","台工 33","台工 34","龍門","牛頭","上方手柄"]`（第69行）。
  4. 「下方大手輪」殘留檢查：確保刨床簡報未誤植搪床／車床的手輪描述（第90-105行）。

### 已完成的驗收紀錄
- 搪床簡報歷經4輪 `DONE → VERIFIED FAIL → DONE → VERIFIED PASS` 反覆，最終於2026-05-27T10:30通過 `VERIFIED|PASS`（`AGENT_SIGNAL.log` 第1-11行）。
- 刨床簡報經Codex產出、Tech Lead校驗後一度 `VERIFIED|Claude|FAIL`（12:27:07，因進度條命名偏離規範、地圖漏標台工34號），熱修復後於12:44:00通過 `VERIFIED|PASS`（`AGENT_SIGNAL.log` 第12-17行；`walkthrough.md` 第70-73行）。
- 鑽床：`AGENT_SIGNAL.log` 全文查無任何鑽床相關 `REQUEST`／`DONE`／`VERIFIED` 紀錄，代表鑽床從未進入過驗收流程（`_context/INDEX.md` 第11、120行）。

### 缺口
- 上述兩支驗收腳本**僅驗證「簡報投影片文字內容」本身**（頁數、禁詞、必要字詞），性質是內容產出的品管工具，**並非「互動機台實體」的驗收計畫**。遍查全repo，找不到任何感測器校準標準、互動操作流暢度測試、機構耐久測試、公眾使用安全驗收（如兒童夾手風險、防護等級）等項目的文件或草稿——**這是七大面向中「驗收計畫」近乎空白的具體證據**。
- 鑽床連「簡報文字」層級的驗收都尚未開始，若三機具要並列交付，鑽床驗收計畫等同從零開始。

## 本章缺口總覽（依嚴重度）
1. **電源與感應規格全缺**：互動機台完全沒有供電、感測器選型與校準的工程規格，現有素材停留在UI/劇本層級，尚未達可施工的工程圖層級，是「技術規格」一節最大的落差。
2. **驗收計畫僅覆蓋簡報文字，不覆蓋實體機台**：現有 `verify_boring_ppt.py`、`verify_planer_ppt.py` 只做文字品管，鑽床甚至連此層級都未開始，公眾使用安全驗收、耐久測試等項目完全空白。
3. **維運面向完全空白**：查無任何維護、保養、故障排除或耗材更換文件，需業主或廠商從零規劃。
