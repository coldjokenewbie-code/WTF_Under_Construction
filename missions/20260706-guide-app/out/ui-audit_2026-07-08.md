## 與正錨落差

- **正錨 v4「主角必須是內容，不是介面」**：一張真史料照撐滿全螢幕、文字退位、按鈕收成一條。現況（`01-landing.png`）：Hero 頂圖確實用了黑白史料照，立意正確；但緊接著是一整排等寬 dev 工具列（375/680／編輯內容／重置任務）、大量 8-11px 的細碎標籤、淺灰背景資訊卡——介面元件的密度和「工具感」明顯壓過內容，與 v4 的克制背道而馳。

- **正錨 v2b「風格語彙必須能追溯到題材本身」**（齒輪線稿、DRG. No.14、SINCE 1927，每個裝飾都答得出「為什麼非蒸汽工廠不可」）：現況展品縮圖（`02-exhibit-G.png`、`11-route-highlight.png`）大量使用白底 3D 算圖蒸汽機車，換到任何鐵道展覽、任何工業博物館都能直接套用，答不出「為什麼非這裡不可」，也沒有比照 v4 補上出處標註。

- **正錨負錨清單**「漸層、圓角、陰影等預設裝飾無題材理由」＋「影像無出處標註」：現況 `ARPage.tsx` 圖片載入失敗時 fallback 到 `picsum.photos` 隨機網路圖（與蒸汽機車、與台北機廠都無關），且展品縮圖普遍無出處/年代標註，直接命中負錨清單的兩項。

- **正錨整體語彙是「暗色、沈浸式的老工廠氛圍」**（v4 與 v2b 皆為深色調）：現況實測全站 24 張截圖 `body` 背景皆為淺灰 `rgb(244,244,245)`，`.dark` class 從未被啟用；多個元件（任務進度條、拼圖投放區）因此呈現「白底硬套深色底樣式」的低對比、近乎隱形狀態，整體氛圍與正錨承諾的沈浸感明顯脫節——不是「風格不同」，而是設計系統本身沒有真正被啟動。

## 卡點

- App 本身順利跑起（`npm install` → `npm run dev`，實際監聽 **3000** port，非 `playwright.config.ts` 預設的 5173；已改連 3000 測試）。
- 本次沙盒網路環境對 `cdn.tailwindcss.com`／`fonts.googleapis.com` 的請求 100% 失敗（`ERR_TUNNEL_CONNECTION_FAILED`／`ERR_CONNECTION_RESET`），導致每個頁面都拋出 `pageerror: tailwind is not defined`。多數畫面仍大致成形（推測是 `index.html` 內建 `<style>` 撐住基礎樣式），但這證實了應改善清單 #3 的架構風險是真實可重現的，不是測試雜訊。
- 漢堡選單展開的截圖（#16）未能確認選單內容是否正常顯示，建議人工在瀏覽器/真機上覆核。

## 主 session 判讀補記（2026-07-08，優化棒必讀）
- **本稽核截圖是在 Tailwind CDN 載入失敗（沙盒網路擋）下拍的**：17/17 頁 `tailwind is not defined` → 截圖呈現的是「近乎無樣式」狀態。純視覺類發現（元素隱形、opacity 未生效、淺灰主題）可能是 CDN 失效的假象，需重驗；**讀碼類發現不受影響**（dark class 從未啟用、header 寫死「掃描QRcode」、framer-motion 僅一處、CDN 依賴本身）。
- **優化第一增量（先於一切視覺項）＝把 Tailwind 從 CDN 改為本地建置**（vite 生態標準做法）：這同時是 P1 修復（展場離線環境 CDN 掛掉＝全站裸奔）與「讓截圖可信」的前提。完成後**重拍全頁截圖、重驗視覺類 P1**，再繼續其餘項。
- 高品味風險項（主題明暗定調、底圖處理、縮圖與 hero 風格統一）照 MISSION 進 _blockers 待使用者裁決，重驗後附可信截圖。
