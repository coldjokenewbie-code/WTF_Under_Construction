# Quality Guard (品質守衛)

## 1. 修改前：三秒原則 (Root Cause First)
- **禁止盲目修補 (Monkey Patch/Hack)**。
- 動手前先確認：(1) 根本原因是什麼？ (2) 這樣改會不會讓架構變複雜？有沒有更簡單的解法？
- 若不確定根本原因，先向 User 提問，禁止直接「試試看」。

## 2. 硬性結構限制 (Hard Limits)
- **函式長度**：單一函式 ≤ 50 行。超過必須抽取共用邏輯。
- **檔案長度**：單一檔案 ≤ 300 行。超過必須提議拆分。
- **命名規範**：
  - 變數/函式使用**台灣慣用英文** (如: `get`, `update`)。禁止單字母命名 (迴圈 `i, j` 除外)。
  - 布林值強制加上前綴：`is_`, `has_`, `can_`。
- **錯誤處理**：禁止無意義的 `try-catch` 掩蓋錯誤。

## 3. 開發與防呆機制 (Self-Healing 熔斷)
- **TDD 思維**：先確認預期行為 (或寫測試) -> 實作 -> 重構。確保每次改動都有驗證點。
- **連續失敗熔斷 (Critical)**：
  - ❌ **失敗 1 次**：停止盲試，分析根本原因並說明理由。
  - ❌ **失敗 2 次**：強制停止修改。向 User 報告目前架構可能的問題與解決方案 (方案 A/B)，等待 User 裁決。

## 4. 驗證與交付 (Verification)
每次修改完成後，回報必須包含以下資訊：
1. **Lint & 測試**：語法無錯誤才算完成。
2. **前端必備**：若是 UI 改動，必須主動提議或說明如何在不同寬度下確認不跑版。
3. **交付格式**：
   > ✅ **修改完成**：[一句話總結] | **根本原因**：[解決了什麼] | **品質**：函式最長 X 行

## 5. 防失憶機制
若對話超過一定長度，AI 須在適當時機的主動回覆開頭加上 `[Quality Guard 啟用中]`，以示未偏離核心規範。
