建議流程

  1. Codex 先建立討論檔，例如 _context/AI_TEAM_DISCUSSION_YYYY-MM-DD_topic.md。
  2. 討論檔放完整背景、資料路徑、問題、回覆格式與 Codex 初始立場。
  3. 先呼叫 Antigravity，scope 限定只讀指定檔案、只 append 指定討論檔與 signal。
  4. 呼叫 Claude 時，優先使用「只輸出文字，不讀寫檔」模式。
  5. Codex 讀取各 agent 回覆後，親自整理進討論檔與最終文件。
  6. 寫入 AGENT_SIGNAL.log 作為流程紀錄。
  7. 若 agent 卡住超過 90 秒，查程序並停止，改用短 prompt 或檔案中繼。

Claude CLI 正式規格

補充者：[Claude@comaMacBookAir] 2026-06-08

登入

  1. 在使用者 Terminal（非 Codex 沙盒）執行：claude /login，瀏覽器完成 OAuth。
  2. 登入後驗證：claude -p "請只回覆 OK"，若回 OK 即成功。
  3. Codex 沙盒若仍回 Not logged in：讓使用者「允許（沙盒外核准）」該指令執行一次，Codex 即可繼續呼叫。token 本身不需要重新登入；問題是沙盒執行權限，不是認證失效。

短 prompt 原則

  • 使用 claude -p "<prompt>"（非互動，單一輸出後結束）。
  • 背景資料直接貼進 prompt 文字，不靠 Claude 自行讀取檔案。
  • prompt 結尾明確標注輸出格式（## Recommendation / ## Risks 等），Tech Lead 直接複製。
  • prompt 過長（> 2000 字）時，先存 /tmp/claude_prompt.txt，用 claude -p "$(cat /tmp/claude_prompt.txt)" 呼叫，避免 shell 轉義斷行。

避免直接讀寫檔

已知問題：要求 Claude 在 CLI 模式下讀取或 append 檔案，會因等待 stdin/stdout 確認而卡住，90 秒後 timeout。

  • 禁止在 prompt 內加入「讀 _context/xxx.md」「append 到 討論檔」等指令。
  • 正確模式：背景文字貼進 prompt → Claude 只輸出結構化文字 → Tech Lead（Codex）寫入檔案。
  • 若背景檔案很大：cat <file> 管入 prompt 字串（見 prompt 範本）。Tech Lead 自行控制截斷長度。

卡住處理（Claude 專項）

症狀處理
Not logged in 使用者 Terminal 執行 claude /login，再用沙盒外核准讓 Codex 呼叫。
90 秒無輸出 Ctrl+C，確認 prompt 是否含讀寫檔指令，改用純輸出模式再試。
輸出中斷（截半） prompt 末尾加 請分段回覆，每段不超過 400 字，或改用 /tmp 檔案中繼。
反覆失敗 改用最短 prompt 驗證（claude -p "請只回覆 OK"），確認工具本身是否正常，再逐步加回背景。

Antigravity CLI 正式規格

由 Antigravity 本人補充｜最後更新 2026-06-08

基本指令與模式

  • 互動模式：agy -i "<prompt>"，啟動一個互動式 Session。
  • 非互動（列印）模式：agy -p "<prompt>"，執行單一 Prompt 並印出結果後結束。
  • 繼續對話：agy -c，繼續最近一次的對話。
  • 指定模型：agy --model <model_name>，指定執行模型。

權限需求與安全

  • 當由其他 Agent（如 Codex）在 Sandbox 內透過 CLI 呼叫 agy 時，若 agy 需要進行檔案讀寫或執行終端指令，會觸發系統的權限請求。
  • 自動授權參數：可在指令後加上 --dangerously-skip-permissions 以自動同意所有權限請求（適用於信任的內部自動化腳本），避免因等待使用者授權而導致 CLI 卡死 Timeout。

寫檔限制與最佳實踐

  • 限制：在 CLI 非互動模式下，直接寫入檔案可能因權限確認或 Sandbox 限制而卡住（超過 90 秒 Timeout）。
  • 最佳實踐：優先使用「只輸出結構化文字，不讀寫檔」模式，讓調用方（Tech Lead）讀取輸出後，再由調用方落檔。
  • 若必須由 agy 寫檔，請提供絕對路徑，並在 Prompt 中限定只修改單一特定檔案。

編碼風險與防範

  • 亂碼風險：跨平台或透過 pipe (|) 轉義輸出時，可能因 Locale 設定不一致產生非 UTF-8 亂碼。
  • 防範：在呼叫 agy 前，確保 Shell 環境變數已設定 export LANG=zh_TW.UTF-8 或 export LANG=en_US.UTF-8。

Signal 寫入慣例

  • 配合 Task Signal Protocol，完成指派任務後，應主動向 AGENT_SIGNAL.log 寫入訊號。
  • 討論訊號格式：DISCUSSION|Antigravity|<相對路徑檔案>|<YYYY-MM-DD>
  • 任務完成訊號格式：DONE|Antigravity|<相對路徑檔案>|<Timestamp>

Codex CLI 正式規格

由 Antigravity 整理補充｜最後更新 2026-06-08

基本指令與模式

  • 互動模式：codex "[PROMPT]"，進入互動式 TUI。
  • 非互動（執行）模式：codex exec "[PROMPT]" < /dev/null 或 codex e "[PROMPT]" < /dev/null，執行單一任務並輸出結果。

重導向防止卡死（核心）

注意：在 headless 模式中執行 codex exec 時，必須使用 < /dev/null。否則 Codex 會因為偵測到 shell 互動式 stdin 而等待額外輸入，導致指令卡死在 Reading additional input from stdin...。

寫檔限制與最佳實踐

  • 限制：直接操作檔案讀寫容易受到 sandbox 權限設定（如 workspace-write）與使用者審查的干擾，增加 headless 模式的 Timeout 風險。
  • 最佳實踐：在 Prompt 中限定為「只輸出結構化文字，不讀寫檔」，由調用方的 Tech Lead（如 Antigravity）攔截輸出，再以專屬工具寫入對應的測試檔或討論檔中。
