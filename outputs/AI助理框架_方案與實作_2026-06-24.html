三個建置層級方案（請 PO 選）

三方討論收斂出同一骨架（本地確定性核心 + 薄 AI 接口 + 安全閘 + 學習迴圈），差別在「建多少」。

方案 A

就地擴充 WTF · 最省
  • 用既有 hook / skill / lessons / nightly 延伸
  • 加「本地優先路由查表 + lesson→skill 升級管線」
  • ✅ 最快、風險最低
  • ⚠️ orchestration 仍半自動（靠 session 觸發）
已實作 · 推薦

方案 B

獨立本地 Orchestrator · 中度
  • 本地 Python assistant/：Broker + Registry + Adapter + Policy Gate + 學習器
  • 任務用 Spec 描述、本地確定性執行為主
  • ✅ 中立/自主/省API/可學習皆到位，可分階段
  • 本次已做出 MVP 並通過驗收（見下）

方案 C

全自動事件驅動 daemon · 重
  • B + 常駐 daemon + 任務佇列 + 跨機自動派工
  • + 複利成本 dashboard
  • ✅ 最自主
  • ⚠️ 建置/維運/安全成本最高

核心設計：五大支柱（皆已用實測證據驗證）

AI 中立
adapter.py 把「呼叫 LLM」抽象成統一介面，底層 claude / codex / agy 可換； 各 AI 只是行為引擎、不持有任務真相源。
證據：測試證實同一介面下三家 argv 正確抽換。
自主執行
一道 run_task.py <spec> 從分類→路由→執行→產物全自動，無人介入。
證據：project_digest 自動掃 10 個 repo 出 HTML 摘要。
省 API 額度
本地優先：命中 registry 就跑確定性腳本、完全不呼叫 LLM；只有未本地化/失敗才 fallback，且預設 dry（不花錢）。
證據：llm_calls_per_task = 0.0
低風險
Policy Gate 雙層 + 受控 executor：拒 網路操作 / rm -rf / git push·clone·pull / 注入字元 / 路徑穿越 / symlink 改名 / 寫入區執行檔；只能寫白名單目錄、永不 shell=True。
自動學習
每步寫結構化事件 → learn.py 算複利指標、偵測「重複 fallback」→ 自動產升級草案（待 PO 審核，不自動生效）→ 升級成 handler 後該類任務 llm_calls→0。
證據：summarize_pdf 連 2 次 fallback 被偵測，自動產 .draft。
資料流：任務 Spec → 分類 → 命中 registry？ 是 → 本地確定性 handler（llm_calls=0）→ Policy Gate 每步攔截 → 產物 ｜ 否 → Fallback Adapter（dry 預設不花 API，帶已完成軌跡避免重做）→ 寫事件 → learn 偵測重複 → 升級草案 → PO 審核 → 本地化（下次零 LLM）

項目結果

怎麼用

python3 tools/assistant/run_task.py tools/assistant/tasks/project-digest.task.json   # 跑任務（零 LLM）
python3 tools/assistant/tests/test_safety.py     # 自驗 30 項（離線）
python3 tools/assistant/learn.py                 # 算複利指標 + 產升級草案
python3 tools/assistant/run_task.py --dump-registry            # 看能力契約
# fallback 預設 dry 不花 API；要真呼叫 LLM 才加 --allow-llm --backend codex|claude|agy
「教它一類新工作」= 在 handlers/ 寫一個 run(spec,ctx) + 在 registry 註冊。 通常由 learn 的升級草案觸發：某類任務一直靠 LLM → PO 核准草案 → 本地化 → 之後零 LLM。這就是越用越省的複利。
