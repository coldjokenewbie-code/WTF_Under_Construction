AI Team — 三方 AI 協作框架

讓 Claude、Codex、Antigravity 三個 AI 工具在同一個專案上分工協作：一個當「指揮」，其餘當「執行」，可即時互相呼叫、派工、驗收。本頁說明它的設計概念、怎麼跑、什麼時候用，並附上可直接複製測試的素材。

產出：Claude@DESKTOP-7SF21LR · 2026-06-08 · 來源：WTF ai-team skill ＋ tools/ai-team/cli-reference.html（出勤專案實測）

★一句話總結

給沒時間看完的人

三個 AI 各有專長且都有命令列（headless CLI），所以可以在同一台機器上由其中一個當 Tech Lead（指揮層），用一行指令直接呼叫另外兩個當 Execution（執行層），把「討論 → 派工 → 收回 → 驗收」串成一條流水線。誰當指揮是動態指派的，依當下哪個工具最適合決定。

💡 核心價值：單一 AI 的盲點由另一個 AI 補。指揮層掌控「做什麼、為什麼」與最終品質；執行層只在明確 scope 內做「怎麼做」，不亂發揮。

1設計概念

兩個角色層 ＋ 動態指派

🎯 Tech Lead（指揮層 / Orchestrator）

決定「做什麼、為什麼」，對最終品質負責。

  • 需求分析、評估影響範圍、決定執行路徑
  • 撰寫 AGENT_SPEC（任務規格）派工
  • 架構 / 資料 / 跨檔 / 診斷類自己做
  • 收回執行層成果後做驗收審查

🔧 Execution Agent（執行層）

在指揮給定的 scope 內做「怎麼做」。

  • 嚴格依 spec 執行，不超出 scope
  • 純視覺 / 單一元件 / 有參照可複製的任務
  • 不做架構判斷；有疑問擱置回報，不自己猜
  • 填充指揮提供的資料，不自行發明 pattern

動態指派：誰當指揮不是固定的

預設 Tech Lead 是使用者本人。也可在對話開頭一句話指派某個 AI：例如「指派 Codex 擔任 Tech Lead」。被指派者升為指揮，其餘兩個（含原本的 Claude）降為執行層。三者完全對稱——任何一個都能當指揮、驅動另外兩個。

Claude

codebase 推理、跨檔重構、文件產出。Claude Code 本身就是 headless CLI。

Codex

沙盒執行、嚴謹步驟拆解。codex exec 驅動。

Antigravity

視覺實作、IDE 整合。agy --print 驅動。

三條設計原則（為什麼這樣設計）

  • 權責分離降低出錯 — 理由 單一 AI 既決策又實作，容易 scope 蔓延、順手亂改。把「決策／品質」收在指揮層，「實作」限制在執行層的明確 scope，錯誤可隔離。
  • 規格驅動，不靠口頭 — 理由 派工用書面 AGENT_SPEC（任務、scope、禁止清單、參照），執行層照規格做。避免「猜名稱、改錯主題、發明新樣式」這類實戰踩過的坑。
  • 禁止推測別人的行為 — 理由 各 AI 的 CLI 規格（登入、寫檔限制、signal 慣例）必須由當事 AI 實測後才落檔，不得他方推測補充，否則會把自己的習慣當成對方的。

2執行方式

同機優先「CLI 直驅」；跨機 / GUI 才用「信號檔」

通道 A：headless CLI 直驅（同一台機器，首選）

三個工具都有命令列模式。指揮層直接用一行 Bash 同步呼叫另外兩個、即時拿回答，不需中繼檔、不需常駐監聽。

工具headless 指令關鍵陷阱
Claudeclaude -p "<prompt>"prompt 別要求它讀寫檔，只要求輸出文字，否則卡住
Codexcodex exec "<prompt>" < /dev/null少了 < /dev/null 會等 stdin 卡死
Antigravityagy --print "<prompt>"寫檔權限彈窗會卡；加 --dangerously-skip-permissions

⚠️ 三者 headless 都是單次無狀態（每次新 session）。多輪討論要由指揮每次把脈絡重新餵進 prompt。卡住超過 ~90 秒：砍程序、縮短 prompt、重跑——不要無限等也不要跳過。

通道 B：信號檔（跨機 Windows↔Mac / GUI 模式 / 需持久化）

當 Bash 叫不到對方的 CLI（不同電腦、或對方在 IDE 互動跑），改用專案根目錄的 AGENT_SIGNAL.log 當訊息板：指揮寫 REQUEST，執行層做完寫 DONE，指揮的 monitor 偵測到就接手驗收。

事件信號格式
派工REQUEST|<AgentID>|<spec_path>|<Timestamp>
討論完成DISCUSSION|<AgentID>|<路徑>|<日期>
任務完成DONE|<AgentID>|<修改的主檔>|<Timestamp>
驗收結果VERIFIED|<AgentID>|PASS 或 FAIL|<Timestamp>
有疑問QUESTION|<AgentID>|<任務>|<問題>|<Timestamp>

完整任務生命週期

1
PO（使用者）下需求

指派或預設 Tech Lead。

2
Tech Lead 需求分析

釐清 what & why、評估影響範圍、決定執行路徑。

3a
分支 A：指揮自己做

架構 / 資料 / 跨檔（>3 檔）/ 診斷類——不下放。

3b
分支 B：下派執行層

純視覺 / 單一元件 / 有明確參照 → 撰寫 AGENT_SPEC → 派發（CLI 直呼或寫 REQUEST 信號）。

4
執行層執行

在 scope 內完成；有疑問擱置回報，不自決。完成回傳 handoff report / 寫 DONE。

5
Tech Lead 驗收審查

檢查：diff 只動 scope 內檔案？有無引入新設計語言？編譯通過？

✓
PASS → commit 並通知 PO ／ FAIL → 退回或指揮接手修正

依「誰當指揮」看實際指令

點選誰是 Tech Lead，下方顯示它驅動另外兩個的實測指令（出勤專案 2026-06-08）。

✅ 出勤專案實測通過。

呼叫 Claude

claude -p "$(cat /tmp/claude_prompt.txt)"

沙盒內若 Not logged in：使用者 Terminal 跑 claude /login，再沙盒外核准執行一次。禁止 prompt 要求 Claude 讀寫檔。

呼叫 Antigravity

agy --print "$(cat /tmp/agy_prompt.txt)"

可讓它直接 append 指定檔，但 prompt 內務必明確限制 scope。中文檔曾出現非 UTF-8 位元組，收回後先跑清理腳本。

✅ 出勤專案實測通過。

呼叫 Claude

claude -p "<prompt>" < /dev/null

< /dev/null 防 stdin 阻塞。只要求輸出文字。

呼叫 Codex

codex exec "<prompt>" < /dev/null

< /dev/null 必加，否則卡在 Reading additional input from stdin...。

ℹ️ Claude 當指揮尚未完整實測，以下為依其他 agent 實測的預期做法，使用前先驗證。

呼叫 Antigravity

agy -p "<prompt>"

權限彈窗加 --dangerously-skip-permissions。只要求輸出文字，Claude 統一落檔。

呼叫 Codex

codex exec "<prompt>" < /dev/null

< /dev/null 必加防 stdin 阻塞。Claude Code 透過 Bash tool 呼叫即可。

3適用場景與理由

什麼時候該下派、什麼時候指揮自己做

分工矩陣

任務類型誰做理由
架構決策（資料結構、路由、型別）指揮影響全局，不可下放
types.ts / 介面定義指揮牽一髮動全身
資料檔案 data/*.ts指揮需領域知識驗證，下放會猜錯內容
跨多檔邏輯修改（>3 檔）指揮執行層缺乏全局上下文
Bug 診斷指揮需讀懂根因，不只症狀
config / CLAUDE.md / AGENT_SPEC指揮定義執行層邊界，不可讓它自訂
單一元件 CSS / 視覺實作執行層範圍明確，影響可隔離
新頁面範本（遵循既有 pattern）執行層有參照，複製延伸即可
純 HTML/CSS 靜態頁執行層無跨檔邏輯
把指揮給的資料填進 UI執行層資料已由指揮確認，只負責呈現

👍 適合用 AI Team 的場景

  • 任務可切成「決策」＋「明確 scope 實作」兩段
  • 有多個視覺元件 / 頁面可平行下派
  • 想要第二個 AI 對方案提意見（討論收斂）
  • 需要互相驗收、降低單一 AI 盲點

👎 不適合 / 沒必要的場景

  • 小改動、單檔、自己幾分鐘搞定——派工開銷更大
  • 需要深度業務邏輯、整體上下文才能做對
  • 探索性除錯（scope 一直變，無法寫死 spec）
  • 純跨機非同步交棒：更新 INDEX/TaskLog 即可，不必開常駐 monitor

原則：猶豫時，預設由指揮自己做。下派的前提是 scope 能寫清楚、有參照可複製。

5素材：讓你快速測試協作

展開即可複製。同事可拿這些直接在自己的專案跑一次三方協作。

① 5 分鐘快速測試（同機，CLI 直驅）

前提：本機已安裝並登入 claude、codex、agy 三個 CLI 其中至少兩個。以下示範你（或 Claude）當指揮，呼叫另外兩個對同一問題給意見：

# 1) 呼叫 Codex 給意見（注意結尾 < /dev/null）
codex exec "用繁體中文，針對『要不要把設定檔從 JSON 改成 YAML』給三點建議，只輸出文字不要讀寫檔" < /dev/null

# 2) 呼叫 Antigravity 給意見
agy --print "用繁體中文，針對同一主題給三點建議，只輸出文字"

# 3) 你（指揮）彙整兩邊意見，做決定、落檔

測試重點：感受「指揮收集多方意見 → 自己拍板」的流程。卡住就照上面「執行方式」的陷阱欄排查（多半是少了 < /dev/null 或要求了讀寫檔）。

② 派工 prompt 範本（呼叫執行層做事）

你是 Antigravity，被納入一個 ai-team 協作，Tech Lead 是我。
請依序：
1. 讀任務規格：<ABS_PATH>/_context/AGENT_SPEC_xxx.md
2. 在 scope 內完成實作（只改 spec 列出的檔案，不改其他）
3. 完成後在 <ABS_PATH>/AGENT_SIGNAL.log append：
   DONE|Antigravity|<你改的主檔>|<時間戳>
有疑問不要自己決定，列在回報的「待確認」區塊。
只做這些，完成後回報你改了什麼。

② 討論 prompt 範本（只要意見、不動檔）

請用繁體中文台灣用語，針對 <任務主題> 給 Tech Lead 意見。
背景：
- <環境 / 資料來源 / 架構>
卡關：
- <卡關點>
請只輸出四段：## Recommendation / ## Detailed Steps / ## Risks / ## Validation
不要讀寫檔。

③ AGENT_SPEC 任務規格範本

指揮派工給執行層時填這份。「禁止修改」清單是關鍵——少了它，執行層容易 scope 蔓延。

# AGENT_SPEC — [任務名稱]
> 建立者：Tech Lead ｜ 執行者：[Antigravity / Codex]

## 任務說明
[1-2 句說做什麼，不說為什麼]

## Scope（只動這些檔案）
- `components/XxxComponent.tsx`
- `components/XxxComponent.css`

## 禁止修改
- types.ts 或任何型別定義
- data/*.ts 資料檔
- CSS 變數（--color-* 等）
- 路由 / config / CLAUDE.md / settings.json
- Scope 以外的所有檔案

## 設計系統約束
- 背景色：[#xxxxxx]　強調色：[#xxxxxx]
- 字型：[主字型] / [mono 字型]
- Panel 樣式：參照 .[class-name]

## 參照元件（「看這個來做」，別自行發明 pattern）
- 結構參照：components/ExistingComponent.tsx
- 樣式參照：index.css 的 .glass-panel

## 預期產出
- [ ] [功能 / 視覺條件 1]
- [ ] [功能 / 視覺條件 2]
- [ ] 編譯無錯誤　- [ ] 無修改 scope 外檔案

## 有疑問時
不要自行決定，列在回報「待確認」區塊，由 Tech Lead 決定。
