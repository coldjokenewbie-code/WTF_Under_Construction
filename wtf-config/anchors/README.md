# 錨點庫（品質判斷的真相源）
> 品味不能寫成規則，只能寫成「對照＋一句話差異」。弱模型做絕對評分會灌水，做對照比較堪用——所以本庫只允許比較。

## 評分鐵律（2026-07-06 立，源自實錯：自評 80 分、實際 10 分）
1. **分數只能產自「算繪截圖 vs 錨點」的並列比較**。流程：Playwright 截圖 → 與本庫錨點並排 → 寫比較句（「相比錨點 A，本作 X 較弱因為…」）→ 依比較句給等第。
2. **無截圖＝無分數**。用文字描述自己的產出然後打分＝無效，該分數視同 0。
3. 等第只有四級：**低於負錨／介於兩者／達正錨／超越正錨**。禁止百分制（百分制是灌水的溫床）。
4. 「達正錨」以上才可交付；宣稱達標必附截圖檔案路徑供人複驗。
5. 手勢、觸控、實機效能：無頭測試只算煙霧測試，一律標「待實機驗」，不得計入達標。

## 打回→錨點更新規則（小隊的學習機制）
使用者每次打回或在多版本中挑選，執行者必須在**同一個工作階段內**：
1. 把落選/被打回的版本與獲選版本（或打回理由）整理成一對新錨點，追加到本庫對應檔；
2. 一句話判準寫「差異在哪」，禁止寫「使用者偏好」（那是拒絕學習的遁詞）；
3. 若打回理由與既有錨點矛盾，保留兩者並標「⚠️待使用者裁決」。
不寫錨點就結束工作階段＝違規（session-end 檢查項）。

## 檔案組織
- 每種交付物一檔：`展示頁面.md`、`報告書.md`、`配樂.md`…（隨任務自然生長）
- 每條錨點格式：`### 錨-編號｜正例 vs 反例（各附檔案路徑/截圖）｜一句話判準`
