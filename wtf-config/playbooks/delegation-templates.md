# 任務交辦 prompt 範本
> 適用：主對話派 subagent（Agent tool）時直接複製填空。`｛｝`＝必填，`（）`＝視情況
> 配套：門檻與 model 選擇見 `playbooks/model-dispatch.md`；驗收判準見 `playbooks/judgment-rubrics.md`
> 通則：每份範本三段固定——目標與動機／驗收條件／回報格式。派工時三段都要填，缺就補完再派

## 通用回報合約（每份範本結尾都附上這段，原文照貼）

```
回報要求：
- 第一行＝一句話結論（完成／失敗／部分完成＋主因）。
- 只回結論與判斷，每個事實附 檔案:行號。禁止貼大段原文。
- 長產物（>50 行）存檔到 ｛存放路徑，預設 outputs/_shared/｝，回傳路徑。
- 回報總長 ≤120 行。
- 失敗就寫失敗＋卡點＋已試過什麼，不要硬掰。
```

## T1. 搜尋／定位（agent: Explore，model: haiku 或 sonnet）

```
在 ｛repo/目錄路徑｝ 找 ｛要找什麼：定義/用法/設定/資料流｝。
動機：｛我拿到結果要做什麼，例：之後要改它，需要知道所有呼叫點｝。
範圍提示：（可能在 ｛目錄/檔名模式｝；已知線索：｛關鍵字/函式名｝）
驗收條件：
- 列出全部命中位置（檔案:行號），漏一處算失敗。
- 每處一句話說明它是什麼角色。
- 若一處都沒找到，回報「用了哪些搜尋詞、哪些目錄」以證明找過。
```

## T2. 實作（agent: general-purpose，model: sonnet）

```
目標：｛做出什麼，一句話｝。
動機：｛為什麼要做、給誰用——遇到規格沒寫到的情況，照這個動機取捨｝。
規格：
- ｛輸入/輸出/行為 逐條列｝
- 遵守專案慣例：檔案放 ｛路徑｝，命名照 ｛慣例或參考檔｝。
- 禁改清單：除 ｛allowlist 檔案/目錄｝ 外不得改動任何檔。
驗收條件（逐條可勾）：
- ｛條件1，例：X 檔存在且 python3 x.py --check 回傳 0｝
- ｛條件2，例：既有測試全過：跑 ｛測試命令｝｝
- 每條驗收附「驗證命令＋實際輸出摘要」，只寫「已確認」不算。
```

## T3. 重構（agent: general-purpose，model: sonnet）

```
目標：把 ｛目標檔/模組｝ 重構為 ｛目標狀態｝，**外部行為不變**。
動機：｛例：重複邏輯三處，之後要加功能會改漏｝。
安全網（先做再動手）：
- 先跑 ｛測試/驗證命令｝ 記下基準輸出；沒有測試就先寫「行為快照」（輸入→輸出對照表）存 ｛路徑｝。
- 小步走：每完成一個獨立步驟就重跑驗證，壞了立刻回退該步，不累積未驗證的改動。
禁改清單：不動 ｛公開介面/檔案清單｝。
驗收條件：
- 重構後 ｛測試/快照｝ 與基準完全一致（附前後輸出對照）。
- diff 只含 ｛允許的範圍｝，git diff --name-only 逐行核對。
- 行為有任何刻意改變＝失敗，回報而不是順手改。
```

## T4. 研究（agent: general-purpose，model: sonnet；高風險結論升 opus）

```
問題：｛要回答什麼，一句話＋判斷成功的樣子｝。
動機：｛這答案要支撐什麼決策｝。
方法要求：
- 來源優先序：｛官方文件/repo 內檔案/實測｝ > 部落格轉述；每個結論標來源（URL 或 檔案:行號）。
- 網路連結必須實際開啟讀過才能引用（GLOBAL.md 規定），只出現在搜尋清單的不算。
- 查不到就寫「查不到」＋查過哪裡，禁止用訓練記憶補——特別是版本號、型號、參數名。
驗收條件：
- 回答問題本身（不是丟資料）；分「確認的事實／推測（標註）／查不到」三欄。
- 關鍵結論至少兩個獨立來源，或一來源＋一實測。
```

## T5. 審查（agent: general-purpose，model: opus；機械檢查可 sonnet）

```
審查對象：｛檔案路徑清單 或 git diff 範圍｝。
你是 fresh-context 審查者：不要假設作者是對的，你的任務是**找碴**，找不到問題也要證明找過。
審查維度（逐項過）：
- 對照驗收條件：｛原任務的驗收條件逐條貼上｝——每條標 PASS/FAIL＋證據。
- 引用完整性：檔內引用的路徑/命令/工具名，逐一 ls/實測確認存在。
- 矛盾檢查：與 ｛相關檔案，例：GLOBAL.md、同目錄其他規則｝ 有無互相打架。
- ｛特定關注點，例：弱模型會不會誤讀這句？型號是不是查證值？｝
驗收條件：
- 輸出問題清單：每條＝嚴重度(P0/P1/P2)＋檔案:行號＋問題＋建議修法。
- 沒問題的維度明寫「查過，無發現」，不可整段跳過。
```
