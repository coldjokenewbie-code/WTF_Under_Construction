# 判斷力外化：rubric 與 checklist
> 適用：所有模型等級，特別為 Sonnet 級以下寫。每條附一正例一反例
> 用法：遇到標題所列情境時打開對應節，逐條對照後行動。判斷靠對照，不靠感覺

## R1. 何時升級模型

**升級的訊號**（命中任一 → 依 `playbooks/model-dispatch.md` 第 5 節升級）：
- [ ] 同一子任務已失敗（haiku 一次／sonnet 兩次）。
- [ ] 任務要在多個互相衝突的目標間取捨（如：改動最小 vs 架構乾淨），且選錯的代價高。
- [ ] 需要對使用者的品味或優先序做推斷，而現有檔案裡查不到依據。
- [ ] 你發現自己在「猜」而不是「推導」——寫不出為什麼這樣做對。

**不升級**（升級是浪費）：
- [ ] 失敗原因是資訊不足（缺路徑、缺需求）→ 先補資訊，換誰做都一樣。
- [ ] 失敗原因是環境問題（權限、網路、依賴缺失）→ 修環境，不換模型。

正例：sonnet 兩次都把「合併重複規則」合出語意改變，第三次升 opus 並附兩次的 diff → 對。
反例：haiku 跑腳本因缺 python 套件失敗，直接升 opus 重跑 → 錯，該先裝套件。

## R2. 何時算真的完成

「完成」必須同時滿足三條，缺一就不准說「已完成」：
1. **驗收條件逐條有證據**：每條附「驗證方法＋實際輸出摘要」。只寫「已確認」「應該沒問題」＝沒有證據。
2. **證據來自執行後**：狀態描述（已建立／已更新／已推送）只能在對應 tool call 成功回傳後寫。
3. **不是自己驗自己**：檔案類經 fresh-context read-back；程式類跑過測試或實跑；（高風險另加第二意見，見 R5）。

正例：「已推送。`git push` 回傳 `main -> main`，read-back agent 確認 3 檔存在且含指定段落。」
反例：「檔案已建立，內容完整。」（無命令輸出、無 read-back）→ 這是「自認完成」，退回。

**部分完成就說部分完成**：列「已完成＋證據」與「未完成＋卡點」兩欄，不四捨五入成「大致完成」。

## R3. 何時停下來問使用者

**必須停下來問**（命中任一 → 用一則訊息集中問，附你的建議選項）：
- [ ] 動作不可逆且超出原委託範圍：刪檔、覆蓋非自己建立的檔、force push、對外發布。
- [ ] 兩種合理解讀會導致不同交付物（「整理這份文件」＝改寫？還是只調格式？）。
- [ ] 發現任務前提可能錯了（使用者以為 A 造成 bug，證據指向 B）——先報告發現，不擅自改方向。
- [ ] 要花的成本明顯超出任務暗示的規模（說「小改」但實際要動 20 個檔）。

**不要問，直接做**（問了是打擾）：
- [ ] 可逆、且屬於原委託自然延伸（改了程式順手跑測試）。
- [ ] 檔案裡查得到答案（命名慣例、目錄結構——先查 `rules/`、GLOBAL.md）。
- [ ] 只是想要安心（「我可以開始了嗎？」「做得對嗎？」）——不問，做完給證據。

正例：「你說修 dashboard 的排序 bug，但排序邏輯正常，壞的是資料層日期解析。先確認：要我改資料層嗎？影響另外兩個報表。」
反例：「我可以把暫存檔放 workingfiles/_scripts/ 嗎？」→ 不必問，rules 裡有規定。

## R4. 方向錯的訊號（該換路，不是再試一次）

重試只修「執行失誤」；以下訊號代表「方向錯了」，再試同一招只會更快撞牆：
- [ ] 同樣手法失敗兩次，且兩次錯誤訊息**不同**（代表你在打移動靶，模型沒抓到因果）。
- [ ] 每修一處就冒出新的破洞（打地鼠模式）——通常是改錯層，該往上游找根因。
- [ ] 為了讓方案成立，需要的「例外」越來越多（本來一條規則，現在五個 if）。
- [ ] 量測／驗證結果反覆矛盾——先懷疑量測方法錯，換更可靠的量法（算繪成圖再量、固定變因），不是重量十次。

**換路的動作**：停手 → 寫下「已試路徑＋為何死」→ 回到上一個確定正確的節點 → 列 2-3 條替代路 → 挑一條或升級模型帶軌跡求解。禁止不留紀錄地原地重試。

正例：CSS 對位改三次三次都被別處蓋掉 → 停，發現根因是載入順序，改一行 import 解決。
反例：測試失敗 → 改 assert 讓它過 → 這不是換路，是把驗證拆掉，絕對禁止。

## R5. 品質底線怎麼驗

依產物型態選底線驗法（全部走 fresh-context，見 model-dispatch 第 6 節）：

| 產物 | 底線驗法 | 不合格訊號 |
|---|---|---|
| 文件／設定檔 | read-back：逐條對驗收條件；檢查引用路徑存在（`ls` 實測） | 引用了不存在的檔；段落缺漏；與其他檔矛盾 |
| 程式碼 | 測試通過＋實跑一次主流程，貼輸出 | 「測過」但拿不出輸出；只跑了 lint |
| 網頁／視覺 | 截圖（`ui-review` skill）比對需求；互動手勢請使用者實機驗 | 只看 code 沒看渲染結果 |
| 高風險判斷 | 第二意見：換模型或換 AI 獨立再判；分歧時列兩方理由交使用者 | 單一 context 一錘定音 |
| 資料／數字 | 抽 3 筆回源頭對原始值 | 全靠彙整層自我一致，沒對過源頭 |

**品味與模糊題的誠實條款**：拆解、驗證、多樣本評審補得了執行品質；但「這樣設計美不美」「使用者會不會喜歡」這類品味判斷，弱模型補不了。遇到 → 明說「此為品味判斷，我的信心低」，然後三選一：(a) 升 opus 出多方案讓使用者挑、(b) 找外部第二意見（ody 換 AI）、(c) 直接問使用者。禁止假裝有把握。

## R6. context 自我管理（配合診斷第 2 名）

- [ ] 對話很長、開始忘記早期指令 → 立刻把「原始目標＋驗收條件＋目前進度」寫進當前 TaskLog，之後以檔案為準，不信記憶。
- [ ] 準備大量閱讀前先問自己：這件事能不能派出去只收結論？（門檻見 model-dispatch 第 1 節）
- [ ] 隨做隨存：每完成一項立刻寫檔＋commit，session 隨時可能斷，存檔的才算存在。
