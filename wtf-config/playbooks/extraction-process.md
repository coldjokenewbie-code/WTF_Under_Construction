# 萃取流程：從多個實例提煉可重用資產（弱模型七步版）
> 用途：骨架、決策樹、範本這類「從 N 個舊專案提煉共用物」的任務。
> 原理：弱模型由上而下抽象很差，由下而上比較堪用——所以整條流程只做盤點、比較、票決、測試，不做自由發揮的抽象。

1. **盤點**：列出全部實例（檔案路徑逐一列，漏列=後面全錯）。實例少於 3 個→停，不夠萃取，回報。
2. **兩兩比對**：每對實例列「共同做法／相異做法＋各自原因」。機械執行，禁止在此步下結論。可派 haiku 分攤，一對一回報。
3. **票決**：做法出現 ≥3 次→「骨架候選」；1-2 次→「特例候選」。純計數，不加判斷。
4. **反例測試**：每條骨架候選問「有沒有任何實例刻意不這樣做？原因正當嗎？」——正當→降為「選項」（附選用條件）；不正當或無反例→留骨架。這一步是品質關鍵，用 sonnet 以上，每條附出處。
5. **品味類不寫規則**：涉及美感、語氣、体驗的差異，不進骨架，改寫成錨點對（見 `wtf-config/anchors/README.md`）。
6. **對抗審查**：fresh-context agent 專查兩種病——過度抽象（骨架裡藏著某專案的特例）、抽象不足（≥3 次的共性被漏掉）。逐條 PASS/FAIL＋出處。
7. **試用驗證**：用產出的骨架起一個假新案，計時並列卡點。骨架的驗收標準是「用得動」，不是「寫得全」。卡點回饋修訂，最多兩輪。

成本預期：單次高階模型的三到五倍棒數。低於此成本的產出多半跳過了第 2 或第 4 步，對抗審查會抓到。
