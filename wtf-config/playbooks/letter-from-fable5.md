> 本檔紅區保存（見 maintenance-protocol.md），有回應另開新檔

## 這套制度是什麼（30 秒版）

使用者用一次高階模型額度，把「判斷力」寫成你能照做的檔案。你不需要比我聰明才能用好它：**路由表告訴你何時開哪個檔，rubric 告訴你怎麼判斷，範本告訴你怎麼派工**。你只要做到一件事——遇到情境時真的去開那個檔，而不是憑感覺。全部入口在 `wtf-config/GLOBAL.md`「制度層」。

## 三件使用者沒問、但我認為最重要的事

### 1. 規則是負債，路由才是資產
這個生態最大的長期風險不是規則太少，是規則太多：每加一條常載規則，之後**每個 session 都付 token 稅**，而且規則一多，弱模型的遵從率反而下降（每條規則都在稀釋其他規則的注意力）。這次重構的核心不是「寫了更多規則」，是把常載鏈從 370 行壓到 241 行、把其餘變成按需路由。**未來請守住這個結構**：新東西預設進 playbooks（按需），只有「每個 session 都需要」的才進常載鏈——而且要先過黃區提案。判斷句：「這條規則如果三個 session 才用到一次，它就不配常載。」

### 2. 別依賴模型自律，把制度變機檢
使用者在 ody 的洞見是對的：**紀律要靠外部結構強制，不是靠模型自我要求**。我寫的 rubric 和鐵律仍然是「靠自律」的文字——這是本次產出的天生弱點。最值得做的下一步（我沒做，留給你們）：把制度裡「可機檢」的部分接進 ody coach 規則庫或 Stop hook，例如：
- 開場檢查：session 前幾個 tool call 若出現逐一 Read `_context/` 多檔 → 違反三檔制，攔截提醒。
- 派工檢查：Agent tool 呼叫的 prompt 若缺「驗收條件」字樣或未帶 `model` 參數 → 提醒補齊。
- 完成檢查：回覆含「已完成」但本輪無成功的 tool 回傳 → 攔截（reply-gate hook 已有類似機制，可加規則）。
文字規則會被遺忘，機檢規則不會。每把一條文字規則轉成機檢，制度就少一個依賴自律的點。

### 3. 便宜模型的真實成本學
「指揮官不下場」省的是主對話 context 和高階額度，但**過度拆分有協調稅**：Haiku 做錯三次（派工×3＋驗收×3＋失敗軌跡整理）常比 Sonnet 一次做對更貴。經驗法則：Haiku 只派「對錯一眼可驗」的活（搜尋、read-back、格式檢查、照範本批次改）；有一點模糊空間的實作直接 Sonnet；**猶豫要不要用 Opus 的時候，通常代表這是判斷點，該用**。另外，使用者有三個不可外包的角色，任何模型都別想替代：黃區核准、實機驗收（手勢/品味）、以及「這個方向值不值得做」的決定。

## 這套制度最可能的退化方式與預防

| 退化 | 徵兆 | 預防（可執行） |
|---|---|---|
| 常載鏈回胖 | 有 session「順手」把新規則塞進 GLOBAL.md | 黃區制度＋門檻：`cat wtf-config/GLOBAL.md wtf-config/CLAUDE_CODE.md wtf-config/AGENTS.md \| wc -l` >500 就抽離（maintenance 第 3 節） |
| playbooks 沒被打開 | 派工沒顯式 model、交辦沒驗收條件＝根本沒讀 dispatch | 使用者抽查最近一次派工 prompt；連續發生 → 把該規則升級成機檢（見上第 2 件事） |
| 矛盾規則再現 | 新檔說 A、舊檔說 B，弱模型選字面明確的那個 | 改規則時必跑 `grep -rn "舊關鍵字" wtf-config/ rules/` 清掉舊句；本次「讀取所有 .md」矛盾就是這樣清的 |
| 驗證儀式化 | read-back 回報永遠 PASS、無證據引用 | 驗證 prompt 一律用 delegation-templates T5（要求找碴＋「查過無發現」逐維度明寫）；PASS 而無 檔案:行號 證據＝無效驗證，重派 |
| 型號表過時被照抄 | 引用 model-dispatch 第 0 節時距查證日 >30 天 | 表頭已標查證日；過期先問 claude-code-guide 再引用，並回填新日期（綠區） |

## 誠實條款：本次產出信心排序（低的先講）

1. **model-dispatch 第 0 節型號表（信心最低）**：查證自 2026-07-03 的 Claude Code remote 環境，使用者本機 CLI 版本可能不同；effort 參數只驗證了「remote 環境 Agent tool 無此參數、Workflow agent() 有」，本機行為未驗。「被導向 Opus 4.8 的請求算誰的額度」完全查不到，已標未確認，建議 usage 儀表板實測。
2. **所有數字門檻（300 行/3 檔/兩輪重試/200 行精簡線）**：是我的判斷值，不是實測值。方向對、數字未必準，用 2-4 週後依實感調（黃區提案改）。
3. **三檔制在非 Claude 工具的落地**：GEMINI.md/CODEX.md/CLAUDE_COWORK.md 文字已改，但那些工具會不會照做未實測（Antigravity 無強制機制，歷史上開場協議就常被跳過——見 ai-degradation.md）。
4. **升降級路徑的觸發精度**：「haiku 錯一次就升級」對某些任務可能過嚴（環境噪音造成的失敗不該算模型的錯——rubric R1 有排除條款，但弱模型可能不會用）。
5. **品味與模糊題**：rubric R5 的誠實條款是極限了——拆解與驗證補得了執行品質，補不了品味。遇到品味判斷，制度只能保證「不假裝有把握」，不能保證做對。

## 給下一個 session 的第一步

現在就做：開 `wtf-config/GLOBAL.md` 讀「制度層」那一節（30 行），把派工鐵律三條記住。之後每次要動手前想一句：「這件事命中路由表哪一行？」命中就開檔照做。沒命中就直接做。就這樣。
