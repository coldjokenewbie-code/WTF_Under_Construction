你如何使用 Claude Code

你是一位動手除錯、迭代精修型的使用者，橫跨多種專案——從 LINE Desktop 自動化（pywinauto）、博物館導覽 App UX，到 HTML 簡報和 Kiosk 應用。你的 session 以除錯與 UI 樣式任務為主（佔前幾大目標中的 19 個），習慣帶著問題進來、中途才加入 Claude，而不是一開始就給完整規格。你常用中文溝通並提供截圖作為視覺參考，但這在 Claude 無法處理圖片時造成了阻礙。Edit-heavy 的工具使用方式（94 次 Edit、90 次 Read）顯示你是在持續微調，而不是大量生成程式碼。

核心模式：你是一位耐挫力高、接受反覆修正的迭代型 debugger，在長時間開著的 session 中跨多個專案作業，聚焦於視覺與 UI 精修。

摩擦點

你的 session 以除錯和反覆修正為主，Claude 頻繁誤解你的意圖或採取錯誤方向，造成大量來回修正。

截圖與檔案處理失敗
你頻繁分享 Claude 無法處理的截圖和檔案，完全阻斷工作流程。建議分享前先調整圖片大小、避免 GIF 格式，或直接貼上錯誤文字。
  • Claude 反覆無法處理 LINE 自動化截圖（超過 API 上限），導致整個 debug session 無法推進
  • Claude 無法讀取 GIF 截圖，要求改貼錯誤資訊，在你期待視覺除錯支援時造成額外摩擦
需求誤解導致反覆重做
Claude 常在第一次就誤解你的意圖——不論是 UI 優先順序、有效資料格式或版面結構——迫使你進入多輪修正循環。事先提供更明確的規格或使用結構化 prompt 能有效減少迭代。
  • Claude 把有效的預填表單 ID 當成垃圾字元清掉，需要你手動糾正
  • Claude 搞錯 Context Map 優先順序（沒把「選擇導覽」設為主功能），也對 Kiosk App 的雙欄卡片結構做出錯誤假設
編輯時過度修改導致壞掉
當你要求針對性修改時，Claude 常常改過頭或引入回歸——壞掉 nav bar、字體改太小，或移除你沒要求改的元素。可以透過要求更小的隔離修改、明確列出哪些東西不能動來緩解這個問題。
  • Claude 把正面標題字體改太小，又沒注意到背面標題還在換行，需要多輪修正
  • Agent 驅動的修改壞掉了行動版 nav bar 並移除了手機外框，被迫 rollback 才能還原

Hooks
在特定生命週期事件（如 pre-commit 或 post-edit）自動執行 shell 指令。
對你的用途：你大量進行 debug 和 UI 樣式調整，需要頻繁的 edit-check 循環。一個在 HTML 編輯後自動提醒的 hook 能減少你在版面驗證上的摩擦。
// .claude/settings.json { "hooks": { "postEdit": { "command": "if echo $EDITED_FILE | grep -q '.html'; then echo '⚠️ 記得確認 nav bar 與版面完整性'; fi" } } }
自訂 Skills
以 Markdown 檔案建立可重複使用的 prompt，透過 /command 觸發。
對你的用途：你已在使用 Skills（4 次呼叫）並建立了 symlink。可以建立 /debug-line skill 給 LINE 自動化流程，以及 /ui-review skill 讓 Claude 在完成前確認 nav bar、字體大小與版面完整性。
// .claude/skills/ui-review/SKILL.md # UI 審查清單 1. 確認所有頁面的 nav bar 可見 2. 確認字體大小不造成文字換行 3. 確認行動版框架/版面完整 4. 測試所有互動元素 5. 列出所有修改的檔案並簡述原因
Task Agents
Claude 生成專注的子 agent 進行平行探索。
對你的用途：你的 debug session（11 次！）常需要探索多種方式——如 LINE 搜尋框輸入問題循環嘗試了多種失敗策略。子 agent 能平行探索替代方案，同時保持主 session 的專注度。
用一個 agent 探索所有可能的方式，將文字輸入到跨程序 Windows GUI 搜尋框（使用 pywinauto），包含 SendKeys、set_text、type_keys、剪貼簿貼上與 UI Automation 模式。回報哪些方式有效、哪些失敗及原因。

事先給足背景，減少除錯來回

未來展望

你的工作流程充滿大量除錯與 UI 迭代循環，這些都可以透過自主、測試驅動的 Claude Code 工作流大幅加速。

測試驅動除錯：自主迭代到通過
你有 11 次 debug session 且頻繁遇到「方向錯誤」的摩擦，可以讓 Claude 自主撰寫捕捉 bug 的測試，然後反覆修正直到通過。這能消除讓你在近三分之一互動中不滿意的來回修正循環。
如何開始：在 CLAUDE.md 中加入測試優先除錯的指示。對你的 TypeScript 和 Python 專案，定義 Claude 可透過 Bash 自主執行的測試指令。
貼到 Claude Code：
我的自動化腳本有個 bug：[描述 bug]。修改前先：1) 寫一個重現並斷言正確行為的測試，2) 執行測試確認失敗，3) 修復程式碼，4) 執行測試確認通過，5) 執行所有現有測試確認沒有回歸。自主迭代直到所有測試通過。步驟之間不需要詢問我確認。
UI 與邏輯平行 Agent
你的博物館 App session 顯示 UI 重構和邏輯修改同時進行時，nav bar 容易被破壞。你可以讓兩個平行 Claude Code agent 分工——一個處理 UI/樣式元件，另一個處理商業邏輯和路由——並以共用規格防止衝突。這完全符合你已有的雙軌開發流程，但讓它更全自動化。
如何開始：以各自作用域的 CLAUDE.md 指示啟動兩個 Claude Code 實例。使用兩個 agent 修改前都會讀取的共用開發規格檔，並讓每個 agent 完成後執行完整 build。
貼到 Claude Code：
你是 UI agent。讀取 /docs/dev-spec.md 了解目前元件結構。你的作用域只有 /components/ui/ 和 /styles/ 裡的檔案。不要修改路由或邏輯檔案。任務：精修行動版 nav bar 版面和字體大小，確保在 375px 視窗下沒有文字換行。每次修改後執行 'npm run build' 確認沒有東西壞掉。完成後列出你修改的每個檔案。
無截圖視覺驗證流水線
多個 session 因 Claude 無法處理截圖而卡死。你可以建立一個自主視覺驗證流水線——Claude 生成 HTML、透過 Bash 啟動無頭瀏覽器、以程式方式擷取並比對截圖、自我修正版面問題——完全不需要手動分享截圖。這能消除你最嚴重的阻礙模式。
如何開始：在專案中安裝 Playwright 或 Puppeteer。Claude 可用 Bash 執行無頭截圖擷取與參考版面的像素級比對。
貼到 Claude Code：
我需要你驗證 /app/index.html 的 UI。自主執行：1) 若未安裝則安裝 Playwright，2) 撰寫腳本在 375px 和 1024px 寬度的無頭 Chromium 開啟 HTML，3) 擷取截圖存到 /test/screenshots/，4) 確認沒有元素超出容器且所有文字在不換行的狀態下可見，5) 若有問題，修復 HTML/CSS 並重新擷取直到版面正確。回報你修了什麼。
